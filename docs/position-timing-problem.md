# 포지션 진입 시점 문제

## 현황

현재 트레이더 포지션 데이터를 `clearinghouseState` API로 가져오고 있음.
이 API는 **현재 보유 중인 포지션의 스냅샷**만 제공하며, 진입 시점(timestamp)을 포함하지 않음.

### clearinghouseState가 제공하는 필드
```json
{
  "coin": "BTC",
  "szi": "119.59754",
  "leverage": { "type": "cross", "value": 40 },
  "entryPx": "66918.2",
  "positionValue": "7954551.98",
  "unrealizedPnl": "-48710.53",
  "returnOnEquity": "-0.2434",
  "liquidationPx": null,
  "marginUsed": "198863.80",
  "cumFunding": {
    "allTime": "-326578.15",
    "sinceOpen": "-10130.42",
    "sinceChange": "81.16"
  }
}
```

**있는 것:** 진입가(entryPx), 크기, 레버리지, PnL, 청산가
**없는 것:** 진입 시점(timestamp), 체결 내역

### 차트에서 발생하는 문제

캔들차트에 트레이더 진입 포인트를 시각화할 때, 진입 시점을 모르기 때문에 "진입가와 가장 가까운 캔들"을 휴리스틱으로 찾아 마커를 배치하고 있음.

```
실제 진입: 3일 전 $67,000에 BTC 롱 진입
현재 표시: 최근 7일 캔들 중 $67,000에 가장 가까운 캔들 → 정확한 시점이 아닐 수 있음
```

결과적으로:
- 마커가 실제 진입 시점이 아닌 엉뚱한 캔들에 찍힐 수 있음
- "최근 진입"과 "며칠 전 진입"을 구분할 수 없음
- 유저 입장에서 "왜 최신 정보가 없지?"라고 느낌

## 사용 가능한 Hyperliquid API

### 1. `clearinghouseState` (현재 사용 중)
- 현재 보유 포지션 스냅샷
- 진입 시점 없음
- 요청당 1명의 트레이더

### 2. `userFills` (미사용)
- 트레이더의 **체결 내역** (매수/매도 히스토리)
- **타임스탬프 포함**
- 최대 2000건 반환 (최신순)
- 요청당 1명의 트레이더

```json
{
  "coin": "ASTER",
  "side": "B",
  "px": "0.65983",
  "sz": "26.0",
  "time": 1743256971233,
  "hash": "0x...",
  "fee": "0.51",
  "crossed": true
}
```

### 3. `userFillsByTime` (미사용)
- 특정 시간 범위의 체결 내역
- startTime, endTime 지정 가능

## 의사결정이 필요한 사항

### 1. 데이터 소스 전략

| 옵션 | 장점 | 단점 |
|------|------|------|
| **A. clearinghouseState만 유지** | 단순, API 호출 적음 | 진입 시점 모름, 차트 마커 부정확 |
| **B. userFills 추가** | 정확한 진입 시점, 체결 히스토리 | 300명 × 추가 API 호출, 레이트 리밋 리스크 |
| **C. 스냅샷 diff 방식** | 추가 API 호출 없음 | Redis에 이전 스냅샷 저장 필요, 첫 수집 이전 포지션은 시점 불명 |
| **D. B+C 하이브리드** | 기존 포지션은 fills로 시점 확인, 이후 변화는 diff로 감지 | 구현 복잡도 높음 |

### 2. userFills를 쓸 경우 고려사항

- **API 호출량**: 300명 × 1 req = 300 req/cycle. 현재 2분 간격이면 분당 150 req
- **Hyperliquid 레이트 리밋**: 공식 문서에 명시적 제한 없으나, 초당 10-20 req이 안전 범위로 알려져 있음
- **데이터 볼륨**: 트레이더당 최대 2000 fills × 300명 = 최대 600,000건. Redis 30MB 무료 플랜에 부담
- **필요한 fills 범위**: 현재 보유 포지션의 진입 시점만 필요하면, 모든 fills가 아니라 현재 포지션의 coin과 매칭되는 가장 오래된 fills만 찾으면 됨

### 3. 스냅샷 diff를 쓸 경우 고려사항

- **Redis 스키마**: `positions:prev:{coin}` 키에 이전 스냅샷 저장
- **diff 로직**: 이전에 없던 포지션이 현재 있으면 → "신규 진입" (시점 = 현재 cron 시각)
- **한계**: cron 시작 이전에 이미 열려있던 포지션은 진입 시점 추정 불가
- **장점**: 추가 API 호출 없음, Redis 사용량 최소

### 4. 차트 UX

진입 시점을 알게 되면:
- 마커를 **정확한 캔들 시점**에 배치 가능
- "N시간 전 진입" 라벨 표시 가능
- "최근 1시간 내 신규 진입" 필터/하이라이트 가능
- 시간순 애니메이션 ("이 트레이더가 먼저 들어가고, 저 트레이더가 뒤따랐다") 가능

## 권장 우선순위

1. **즉시**: 스냅샷 diff 구현 (추가 API 호출 없이 "신규 진입" 감지)
2. **단기**: 상위 S/A-tier 트레이더 한정으로 userFills 호출 (30-50명, 레이트 리밋 안전)
3. **중기**: fills 데이터 기반으로 차트 마커 정확도 개선 + "최근 진입" 하이라이트
